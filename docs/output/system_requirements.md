# システム要件定義書

## システム概要

SoundMapは、旅先や日常の環境音・生活音を30秒間録音し、世界地図上で位置情報と紐づけて共有する音声特化型SNSである。忙しい社会人が抱える「旅行に行きたいが時間がない」「SNS・ショート動画による情報疲れ」という課題に対し、視覚に頼らない"音だけ"の体験を提供する。ユーザーは地図上のピンをタップして目を閉じるだけで、30秒間の「余白の時間」を得られる。30秒固定という制約により時間を浪費させず、ショート動画の逆を行く"引き算"の設計思想を持つ。投稿が蓄積されるほど地図上の音が充実するネットワーク効果と、特定の場所・季節・時間帯でしか録れない音の希少性がMoatを形成する。フリーミアムモデルで収益化し、ウェルネス・マインドフルネス市場の拡大を追い風に成長を目指す。

### 1. システムの目的

- **解決する具体的な課題**
  - 忙しい社会人が旅行に行けず、旅行願望が満たされないまま日々を過ごしている
  - SNS・ショート動画のループにより時間を浪費し、罪悪感と精神的消耗を招いている
  - 寝る前にリフレッシュしたいが、視覚依存のコンテンツしか選択肢がなく、逆に目が冴えてしまう
- **目標とする成果**
  - ユーザーが寝る前に30秒の音声体験でリラックスし、習慣的に利用する状態を作る
  - ユーザーが自発的に環境音を録音・投稿し、コンテンツが自然増殖するエコシステムを構築する
- **期待されるビジネスインパクト**
  - 初期100人→1年後に主要都市・観光地の音が充実しアクティブユーザー拡大
  - フリーミアムによるプレミアムプラン課金収益の確立

### 2. 主な機能

- **コア機能**
  1. 世界地図表示＋音声スポットのピン表示
  2. ピンタップ→投稿情報表示→30秒音声再生（自動停止）
  3. 環境音の30秒録音＋位置情報付き投稿
  4. ユーザー登録・ログイン（メール/SNS）
- **差別化機能**
  1. 30秒固定の音声体験（時間浪費を防ぐ設計）
  2. 地図UIによる探索体験（音を"場所"から発見する）
  3. 再生中の没入オーバーレイ（視覚情報を極限まで削減）
- **付加価値機能**
  1. いいね（保存）＋保存一覧による音のライブラリ化
  2. 高音質再生・オフライン保存等のプレミアム機能

### 3. 提供する価値

- **ユーザーにとっての直接的価値**
  - 目を閉じて30秒、世界中のリアルな環境音でリラックスできる「余白の時間」
  - 旅行に行けなくても「その場にいる感覚」を手軽に得られる疑似旅行体験
  - SNS疲れの悪循環から抜け出す、引き算のデジタル体験
- **ビジネスにとっての価値**
  - ネットワーク効果によるユーザー投稿の自然増加とコンテンツの希少性による参入障壁構築
  - フリーミアムモデルによる段階的な収益化
- **社会的価値**
  - デジタルウェルネスへの貢献（SNS疲れの軽減）
  - 世界中の環境音のアーカイブとしての文化的価値

### 4. 実現方法の概要

- **主要なテクノロジー要素**
  - Next.js（React）によるWebアプリケーション基盤
  - 地図ライブラリ（Mapbox GL JS等）による世界地図と音声スポットの表示
  - Web Audio APIまたはネイティブ録音機能による30秒音声録音
  - Supabase（PostgreSQL）による音声メタデータ・ユーザーデータの管理
  - Supabase Storageによる音声ファイルの保存・配信
- **重要なインテグレーションポイント**
  - Clerk認証との連携（メール/Google/Apple ログイン）
  - 地図API（Mapbox）との連携
  - ブラウザの Geolocation API による位置情報取得
- **スケーラビリティへの考慮**
  - Vercelのエッジネットワークによる低レイテンシ配信
  - Supabase Storageによる音声ファイルのスケーラブルな保管
  - CDN経由の音声配信により、ユーザー増加に伴う負荷に対応

### 置いた仮定

1. **プラットフォーム**: 入力情報ではスマートフォンアプリ（iOS/Android）と記載されているが、技術スタック指定がNext.js（Webアプリ）であるため、**PWA（Progressive Web App）としてモバイル最適化したWebアプリ**で実装すると仮定する。将来的にReact Nativeやネイティブアプリへの展開を検討する。
2. **初期ユーザー規模**: MVP段階で数百人規模（目標100人）、1年後に数千〜数万人規模を仮定。
3. **開発チーム**: 少人数（1〜3名）のチームを想定。フルスタック開発者が主導する前提。
4. **予算**: スタートアップ初期段階。低コストなインフラを優先（Vercel無料枠、Supabase無料枠の活用）。
5. **地図API**: Mapbox GL JSを使用と仮定（Google Maps APIより無料枠が大きく、カスタマイズ性が高い）。

## 技術スタック

### 推奨技術スタック

| レイヤー | 技術 | バージョン |
|----------|------|-----------|
| フロントエンド | Next.js (React) | 16以降 |
| 言語 | TypeScript | 5.x |
| バックエンド | Next.js App Router (API Routes / Server Actions) | — |
| データベース | Supabase (PostgreSQL) | — |
| 認証 | Clerk | — |
| ホスティング | Vercel | — |
| 地図 | Mapbox GL JS (react-map-gl) | — |
| 音声処理 | Web Audio API / MediaRecorder API | — |
| ストレージ | Supabase Storage | — |
| スタイリング | Tailwind CSS | 4.x |
| 状態管理 | Zustand（軽量で十分） | — |
| UIコンポーネント | shadcn/ui（ダークテーマとの親和性が高い） | — |

### 選定理由

1. **Next.js 16 + App Router**: サーバーコンポーネント/クライアントコンポーネントの使い分けにより、地図画面の初期表示速度を最適化できる。App RouterのAPI RoutesでバックエンドAPIも同一プロジェクト内に構築でき、少人数チームでの開発効率が高い。
2. **TypeScript**: 型安全性によりバグの早期発見が可能。音声データや位置情報など多様なデータ型を扱う本プロジェクトでは特に有効。
3. **Supabase (PostgreSQL)**: リアルタイムサブスクリプション（新規投稿の即時反映）、Storage（音声ファイル保存）、Row Level Security（データアクセス制御）を統合的に提供。PostGISによる位置情報クエリも可能。無料枠が充実しておりMVP段階のコスト最適化に寄与。
4. **Clerk**: メール/Google/Apple認証をわずかなコード量で実装可能。Webhookによるユーザー情報のSupabase同期も容易。
5. **Vercel**: Next.jsとの最適な統合。エッジネットワークによる低レイテンシ配信。プレビューデプロイによるレビュー効率化。無料枠でMVP運用可能。
6. **Mapbox GL JS**: Google Maps APIと比較して無料枠が大きい（月50,000リクエスト）。ダークテーマの地図スタイルが標準提供されており、UI方針と合致。WebGLベースのスムーズな地図操作。
7. **Tailwind CSS + shadcn/ui**: ダークテーマのカスタマイズが容易。shadcn/uiはコピー&ペーストベースでバンドルサイズを最小限に抑えられる。
8. **PWAアプローチ**: iOS/Android両対応をワンコードベースで実現。ネイティブアプリのようなインストール体験を提供しつつ、Web技術のみで開発可能。将来ネイティブ化が必要になった場合はReact Nativeへの部分移行も検討可能。

### 技術的な注意点

- **PWAの制約**: iOSのSafariではWeb Audio API/MediaRecorder APIに制約がある場合がある。録音機能のクロスブラウザテストが必要。
- **位置情報**: Geolocation APIはHTTPS環境でのみ動作。Vercelデプロイにより標準で対応。
- **音声ファイル形式**: WebM（Opus）またはMP4（AAC）での録音を想定。ブラウザ互換性に応じてフォーマットを自動選択する仕組みが必要。

## 開発ロードマップ

### Phase 1: MVP（4〜6週間）

**根拠**: コア仮説（「音だけで旅行気分やリラックスを感じられるか」）の検証に必要な最小限の機能に絞る。技術的にも認証→地図→音声再生→録音投稿の順に依存関係がある。

| 週 | マイルストーン | 主なタスク |
|----|--------------|-----------|
| Week 1-2 | 基盤構築 | プロジェクトセットアップ、Clerk認証、Supabaseスキーマ設計、DBマイグレーション |
| Week 2-3 | 地図・再生機能 | Mapbox地図表示、ピン表示・クラスタリング、ボトムシートUI、30秒音声再生 |
| Week 3-4 | 録音・投稿機能 | 録音UI、MediaRecorder実装、プレビュー再生、位置情報取得、投稿API |
| Week 5 | 統合・UI仕上げ | 画面遷移の統合、ダークUI仕上げ、没入オーバーレイ、レスポンシブ対応 |
| Week 6 | テスト・デプロイ | E2Eテスト、パフォーマンス検証、シード投稿作成、本番デプロイ |

**MVP成功基準:**
- 初期ユーザーの50%以上が「また使いたい」と回答
- 7日リテンション率15%以上
- 1ユーザーあたり平均3回以上の音声再生/セッション

### Phase 2: エンゲージメント強化（MVP後 2〜4週間）

**根拠**: MVPで検証されたコア体験の上に、リテンション向上とコンテンツモデレーションの仕組みを追加。ユーザーフィードバックに基づく改善も含む。

| 機能 | 目的 |
|------|------|
| いいね（保存）機能 | 再訪問のモチベーション向上 |
| 保存一覧画面 | 保存した音への簡単なアクセス |
| 投稿履歴 | 投稿者のモチベーション維持 |
| 通報機能 | コンテンツ品質の維持 |
| プッシュ通知（PWA） | リテンション率の改善 |

### Phase 3: グロース・マネタイズ（Phase 2後）

**根拠**: 十分なコンテンツとユーザーベースが蓄積された段階で、発見性の向上とマネタイズ機能を追加。

| 機能 | 目的 |
|------|------|
| エリア・国フィルター | コンテンツの発見性向上 |
| 人気スポット表示 | 新規ユーザーの初回体験改善 |
| 季節・時間帯タグ | コンテンツの文脈情報強化 |
| プレミアムプラン | 収益化（高音質再生、オフライン保存、保存数無制限） |

## リスク分析

### 技術的リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| iOSのPWA録音制約 | 高 | 中 | MVP初期段階でiOS Safariの録音APIの挙動を検証。制約がある場合はCapacitorやTWA（Trusted Web Activity）による部分ネイティブ化を検討 |
| 地図パフォーマンスの劣化（ピン増加時） | 中 | 高 | ピンのクラスタリング、ビューポート外ピンの遅延読み込みを実装 |
| 音声ファイルのストレージコスト増大 | 中 | 中 | 音声の圧縮（Opus/128kbps）、CDN経由配信でのコスト最適化 |
| ブラウザ間の音声フォーマット非互換 | 低 | 中 | サーバーサイドでのフォーマット変換（ffmpeg等）をフォールバックとして用意 |

### ビジネス的リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| コールドスタート問題（ピンがスカスカ） | 高 | 高 | 運営側で主要都市・観光地の音を50〜100件シード投稿する。旅行系インフルエンサーとの連携も検討 |
| 投稿モチベーション不足 | 高 | 中 | 投稿数に応じたバッジ、聴かれた回数の表示、投稿者プロフィールの充実 |
| 30秒体験の物足りなさ | 中 | 中 | ユーザーテストで体験の満足度を定量的に計測。秒数の調整や連続再生モードを検討 |
| フリーミアム転換率の低迷 | 中 | 中 | プレミアム機能の価値検証をPhase 3前に実施。無料ユーザーへのアンケートで課金意向を確認 |

### 法的・コンプライアンスリスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| 個人情報保護（位置情報） | 高 | 低 | 投稿時のみ位置情報を取得。プライバシーポリシーで明示。GDPR/個人情報保護法に準拠 |
| 著作権侵害の音声投稿（音楽等） | 中 | 中 | 利用規約で環境音のみの投稿を明記。音楽検出APIの導入を将来検討 |
| 不適切コンテンツの投稿 | 中 | 中 | 利用規約の整備、通報機能（Phase 2）、将来的なAI音声分類の導入 |

### 運用的リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| 少人数チームでの運用負荷 | 中 | 高 | MVP段階では手動レビューを最小限に。投稿審査のワークフローを自動化 |
| サービス障害時の対応 | 中 | 低 | Vercel/Supabaseの可用性に依存。アラート監視（Vercel Analytics、Supabase Dashboard）を設定 |

## UI/UXの方向性

- **デザインコンセプト**: ダーク・ミニマル・没入型。余計な情報を徹底的に削ぎ落とし、「音」と「地図」だけが主役になるUI。Headspace、Dark Noise、Muji to Sleepの世界観を参考にする。
- **トーン＆マナー**: 静謐で上品。主張しすぎないタイポグラフィと控えめなアニメーション。通知やバッジなどの中毒的要素は排除し、「引き算」の設計思想を徹底する。
- **ターゲットデバイス**: モバイルファースト（スマートフォンでの寝る前利用がメインシーン）。タブレット・PCにもレスポンシブ対応。
- **カラーパレット**:
  - 背景: `#0a0a0f`〜`#121212`
  - メイン: ダークネイビー `#1a1a2e`〜`#16213e`
  - アクセント: 淡いブルーまたはアンバー（ピン・再生ボタン）
  - テキスト: オフホワイト `#e0e0e0`、セカンダリ `#888888`
- **参考サイト/アプリ**:
  - [Headspace](https://www.headspace.com/) — ダークUI、リラックス促進の体験設計
  - [Dark Noise](https://darknoise.app/) — 環境音アプリのシンプルなUI
  - [Google Earth](https://earth.google.com/) — 地図の没入感と探索体験
